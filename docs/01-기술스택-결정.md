# 기술 스택 결정 문서 (Tech Stack Decision)

> 작성일: 2026-07-18
> 상태: **확정**
> 프로젝트: 학습기록(Study Log) 윈도우 데스크톱 앱

---

## 1. 프로젝트 개요

로컬에서 동작하는 **개인용 학습기록 데스크톱 앱**을 만든다.

- **최우선 기능**: 학습 기록(무엇을, 얼마나, 언제 공부했는지 입력·저장)
- **핵심 기능**: 기록을 바탕으로 **일간 / 주간 / 월간 통계** 시각화
- **저장 방식**: 로컬 (서버·클라우드 없음)
- **대상 플랫폼**: 윈도우 (Windows 11)

### 제약 및 목표
- 앱 개발 경험 없음 → **"앱을 완성해보는 것" 자체가 목표**
- **UI 완성도(디자인)에 욕심**이 있음
- 지금은 개인 전용, **추후 일반 공개(배포) 의향 있음**
- 실제 코드 작성은 AI가 담당 → 특정 언어의 난이도는 큰 제약이 아님

## 2. 최종 결정

| 구분 | 선택 |
|------|------|
| 데스크톱 프레임워크 | **Tauri** |

## 3. 검토한 대안들

윈도우 데스크톱 앱을 만드는 방법은 크게 **네이티브(C#/.NET)** 계열과 **웹기술** 계열로 나뉜다.

### 3-1. 전체 후보 비교

| 스택 | 언어 | 장점 | 단점 | 초보 적합도 |
|------|------|------|------|:---:|
| WinForms | C# | 배우기 가장 쉬움, 드래그&드롭 UI, 자료 풍부 | UI가 올드함, 차트 투박 | ⭐⭐⭐⭐⭐ |
| WPF | C# | 현대적 UI, MVVM, 성숙한 생태계 | XAML 학습곡선 | ⭐⭐⭐⭐ |
| WinUI 3 | C# | 최신 윈도우 디자인(Fluent), MS 미래 방향 | 자료 적음, 초기설정 까다로움 | ⭐⭐⭐ |
| Avalonia | C# | WPF 유사 + 크로스플랫폼 | 커뮤니티 상대적으로 작음 | ⭐⭐⭐ |
| **Electron** | JS/TS | 웹기술 그대로, 차트 라이브러리 최강, 예쁜 UI 쉬움 | 앱 용량 큼(100MB+), 메모리 무거움 | ⭐⭐⭐⭐ |
| **Tauri** | Rust+웹 | Electron보다 훨씬 가벼움, 웹 UI 사용 | Rust 백엔드, 초기 설정 하드 | ⭐⭐ |
| Python + PyQt/Tkinter | Python | 빠른 개발, 데이터 처리 편함 | 배포(exe화) 번거로움, **백신 오탐** | ⭐⭐⭐ |
| Flutter | Dart | 예쁜 UI, 크로스플랫폼 | 데스크톱 아직 덜 성숙, 새 언어 | ⭐⭐ |

### 3-2. 탈락 사유

- **Python (PyQt/Tkinter)** — ❌ 탈락
  PyInstaller로 만든 exe가 **윈도우 Defender/백신의 오탐(false positive)**에 자주 걸린다. 실제로 배포하면 사용자가 "바이러스 경고"를 보게 되어 공개 배포 목표와 정면으로 충돌한다.

- **C# 네이티브(WinForms/WPF/WinUI)** — 보류
  정통 윈도우 앱을 제대로 배우기엔 좋지만, (1) UI 자유도·디자인 측면에서 웹기술만큼 자유롭지 않고, (2) 크로스플랫폼 확장 여지가 없다. "예쁜 UI" 욕심과 웹기술 선호를 고려해 제외.

- **Flutter** — 보류
  데스크톱 지원이 상대적으로 덜 성숙하고 Dart라는 새 언어를 도입해야 함. 이점이 뚜렷하지 않아 제외.

- **Electron** — 최종 후보였으나 차점
  아래 4장 참고.

## 4. 왜 Tauri인가 (Electron과의 최종 비교)

웹기술 계열로 좁힌 뒤, 최종 결정은 **Electron vs Tauri** 두 개로 압축되었다.

| 항목 | Electron | Tauri |
|------|----------|-------|
| 앱 용량 | 무거움 (100MB+) | **가벼움 (~5~15MB)** |
| 메모리 사용 | 무거움 | **가벼움** |
| UI 자유도 | 웹 그대로 (최고) | 웹 그대로 (최고) — **동일** |
| 성숙도 / 자료량 | **압도적으로 많음** | 적지만 충분 |
| 백엔드 언어 | Node.js (JS) | Rust |
| 초기 설정 난이도 | 쉬움 | 어려움 (Rust + C++ 빌드도구) |
| 배포/공개 적합성 | 좋음 | **더 좋음** (작고 깔끔) |

### 결정 근거

1. **공개 배포에 유리하다.**
   Tauri 결과물은 수 MB로 매우 작다. "다운로드 받아 쓰는" 앱으로서 인상이 좋고, 배포·업데이트 부담이 적다. 추후 일반 공개 의향과 가장 잘 맞는다.

2. **UI 자유도는 Electron과 동일하다.**
   둘 다 화면은 웹(HTML/CSS/JS)으로 그린다. 따라서 "예쁜 UI" 욕심을 채우는 데 둘 사이 차이가 없다. 그렇다면 더 가벼운 쪽이 유리하다.

3. **Tauri의 유일한 큰 단점(Rust 백엔드)이 이 프로젝트에선 무력화된다.**
   개발을 AI가 담당하므로 Rust 코드 작성 부담을 사용자가 지지 않는다. Tauri의 최대 진입장벽이 사실상 사라진다.

4. **윈도우11은 WebView2가 기본 내장**되어 있어(확인 완료: v150 설치됨) 별도 런타임 설치가 필요 없다.

> **트레이드오프(감수한 것):** Tauri는 자료·커뮤니티가 Electron보다 적고, 초기 개발 환경 설정(Rust + Visual C++ Build Tools 설치)이 번거롭다. 하지만 이 설정은 **최초 1회**만 넘으면 되며, 이후 이점(가벼움·배포 유리)이 이를 상쇄한다.

## 5. 프론트엔드: 왜 React + Tailwind인가

- **React** — 자료와 컴포넌트 생태계가 가장 풍부하다. 특히 `shadcn/ui` 등 완성도 높은 UI 컴포넌트 라이브러리를 바로 쓸 수 있어 "예쁜 UI"를 빠르게 만든다. AI가 다루기에도 학습 데이터가 가장 많아 안정적이다.
- **Tailwind CSS** — 클래스 기반 스타일링으로 디자인 반복이 빠르고 일관성이 좋다. 통계 대시보드처럼 레이아웃이 많은 화면에 유리하다.
- 대안이었던 Svelte/Vue도 훌륭하지만, 자료량·컴포넌트 생태계·AI 친화성에서 React가 우위.

---

## 6. 데이터 저장: 왜 SQLite인가

- 로컬 단일 파일 DB로 **서버가 필요 없다.**
- 학습기록처럼 **날짜별 집계(GROUP BY)**가 핵심인 앱에 SQL이 이상적이다. 일/주/월 통계 쿼리를 표준 SQL로 간단히 뽑을 수 있다.
- Tauri 생태계에 SQLite 플러그인(`tauri-plugin-sql`)이 있어 통합이 쉽다.

---

## 7. 배포 시 유의사항 (미래 참고)

- Electron/Tauri를 막론하고, **코드 서명(code signing)이 안 된 앱은 윈도우 SmartScreen이 "알 수 없는 게시자" 경고**를 띄운다.
- 이는 Python의 백신 오탐과는 별개 문제이며, **코드 서명 인증서**(연 10~20만원대)를 구매하면 해결된다.
- **개인 전용 단계에서는 신경 쓰지 않아도 된다.** 일반 공개 시점에 다시 검토.
